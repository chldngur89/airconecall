# 다른 LLM에 그대로 붙여넣는 프롬프트

아래 아이템을 기준으로 **서비스 기획서, MVP 명세서, 화면 흐름, 운영 정책, 개발 범위**를 실무적으로 정리해줘.

## 역할
너는 스타트업 서비스 기획자이자 O2O 플랫폼 PM이야.  
초기 앱을 크게 만들기보다, 빠르게 시장 검증이 가능한 MVP 관점으로 판단해줘.

## 아이템 개요
에어컨 기사 매칭 서비스다.  
기존 커플 매칭이나 택시 매칭처럼, 사용자가 지금 필요한 시간과 위치를 올리면 근처의 검증된 에어컨 기사와 매칭되는 구조다.

사용자 예시는 다음과 같다.

- 밤 10시에 에어컨이 고장 났다.
- 제조사 AS는 늦고, 숨고 같은 견적 비교 서비스는 기다려야 한다.
- 사용자는 지금 바로 올 수 있는 기사를 원한다.
- 사용자는 대략적인 평균 금액을 먼저 보고 싶어 한다.
- 기사는 우리 회사가 보유하거나 확보할 DB 기반으로 운영한다.
- 초기에는 자동매칭보다 관리자 수동 배차로 시작해도 된다.

## 핵심 차별점
- 숨고처럼 견적을 기다리는 방식이 아니다.
- 카카오택시처럼 지금 가능한 기사와 매칭되는 구조다.
- 에어컨 기사 DB, 가능 시간, 지역, 기술 분야, 장비 보유 여부를 기준으로 매칭한다.
- 고객에게는 평균가 기반 예상금액을 보여준다.
- 현장 추가금은 고객 승인 후에만 청구한다.

## 원하는 산출물
다음 형식으로 정리해줘.

1. 서비스 한 줄 정의
2. 해결하려는 문제
3. 핵심 고객군
4. MVP 범위
   - 포함할 기능
   - 제외할 기능
5. 고객 사용자 흐름
6. 기사 사용자 흐름
7. 관리자 운영 흐름
8. 핵심 기능 목록
   - 고객 기능
   - 기사 기능
   - 관리자 기능
9. 가격 정책 초안
   - 출장비
   - 야간/주말 할증
   - 예상 작업비
   - 부품비/냉매비
   - 추가금 승인 정책
10. 기사 DB에 필요한 필드
11. 운영 정책
   - 취소
   - 노쇼
   - 추가금
   - 재방문
   - 클레임
   - 야간 방문 안전
12. 2주 파일럿 실행안
13. MVP 검증 지표
14. 향후 자동매칭으로 확장할 때 필요한 기능
15. 최종적으로 투자자나 내부 팀에게 설명할 수 있는 1페이지 요약

## 제약 조건
- 처음부터 고객앱과 기사앱을 모두 만들지 않는다.
- MVP는 고객용 모바일 웹, 관리자 콘솔, 기사 문자/카카오톡 링크 정도로 시작한다.
- 초기 지역은 서울 강남·서초·송파 중 1개 권역으로 제한한다.
- 초기 서비스는 가정용 벽걸이, 스탠드, 2in1 에어컨 긴급 점검/수리로 제한한다.
- 대형 시스템에어컨, 외벽 고소작업, 제조사 보증수리는 초기 범위에서 제외한다.
- 답변은 실무자가 바로 실행할 수 있게 구체적으로 작성해줘.
- 마케팅 문구보다 운영 가능성과 검증 가능성을 우선해줘.

## 톤
대표/기획팀/개발팀이 같이 볼 수 있는 문서처럼 명확하고 실무적으로 작성해줘.
